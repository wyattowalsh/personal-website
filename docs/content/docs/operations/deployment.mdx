## Overview

The site is designed for deployment on Vercel with automatic builds from the `master` branch. The project is structured as a pnpm monorepo with two deployable packages: the main site and the documentation site.

## Monorepo Structure

The project uses pnpm workspaces defined in `pnpm-workspace.yaml`:

```yaml
packages:
  - '.'      # Main site (personal-website)
  - 'docs'   # Documentation site
```

### Deployment Configuration

Each workspace can be deployed independently:

| Package | Directory | Build Command | Output Directory |
| :--- | :--- | :--- | :--- |
| Main site | `/` | `pnpm build` | `.next` |
| Docs site | `docs/` | `pnpm --filter docs build` | `docs/.next` |

**Vercel configuration:**

The root `package.json` serves as the default build target. To deploy the docs separately, create a separate Vercel project pointing to the `docs/` subdirectory with build command `pnpm build`.

## Analytics Integration

The site always mounts Vercel Analytics and Speed Insights, and sends anonymous custom interaction events to PostHog when `NEXT_PUBLIC_POSTHOG_TOKEN` is configured. Google Analytics and Google Tag Manager are optional and only load when their public env vars are set.

### Configuration

Analytics are loaded in the root layout and through `components/PostHogPageviewTracker.tsx`:

```tsx
{gtmId && <GoogleTagManager gtmId={gtmId} />}
{gaId && <GoogleAnalytics gaId={gaId} />}
<PostHogPageviewTracker />
<Analytics />
<SpeedInsights />
```

**Why @next/third-parties?**

- **Performance optimized** — lazy loads scripts, uses `next/script` strategy
- **First-party integration** — official Next.js package
- **Type-safe** — TypeScript definitions included
- **No consent banner required** — follows best practices for GDPR compliance

### Tracked Events

The analytics layer tracks:

- **Page views** — route changes captured as anonymous PostHog `$pageview` events
- **Core Web Vitals** — LCP, FID, CLS, TTFB, INP
- **Scroll depth** — public page scroll milestones
- **Time on page** — public page time milestones
- **Content engagement** — reading progress, site search, share clicks, outbound link clicks, code copies, and theme changes

The `/admin` dashboard reads aggregate PostHog data from the server with `POSTHOG_PERSONAL_API_KEY`.

### Privacy Considerations

- Custom PostHog events use random anonymous browser/session IDs
- PostHog person-profile processing is disabled for site events
- `/admin` is excluded from public visitor analytics capture
- Search query analytics are sanitized before capture

## Admin Dashboard & Analytics

### Overview

The `/admin` dashboard aggregates visitor analytics from PostHog and stores time-series rollup data (pageviews, sessions, engagement by dimension) in a Turso SQLite database. The analytics pipeline runs automatically during deployments and cron jobs, building daily and dimension-level summaries for fast querying.

For comprehensive documentation on admin components, architecture, development workflows, and troubleshooting, see [Operations > Admin Dashboard](/docs/operations/admin-dashboard).

### Schema Health & Repair

The admin dashboard includes automated database schema management to handle fresh installations and schema evolution:

**Schema Health States:**

- **`healthy`** — Tables exist with hardened NOT NULL constraints on primary keys (safe to use)
- **`outdated`** — Tables exist but lack NOT NULL constraints (migration needed)
- **`unknown`** — Tables don't exist or schema can't be determined (initialization needed)

**Automatic Safety Mechanisms:**

1. **Fresh Database Initialization**
   - If tables don't exist (`health === 'unknown'`), the schema is created with hardened DDL including NOT NULL constraints
   - No data loss risk — only applies when tables are missing
   - **When:** On first deploy, after database reset, or during recovery from missing tables

2. **Race Condition Handling**
   - COUNT queries (used to check if data exists before repair) are wrapped in try-catch
   - If a COUNT fails (due to concurrent data operations or transient errors), repair treats the table as empty and proceeds safely
   - **When:** During high-traffic analytics writes or concurrent admin operations

3. **Data-Aware Repair**
   - Repair only proceeds if tables are empty (checked via COUNT queries)
   - If tables contain data, repair is blocked with a clear error message recommending manual review
   - **When:** During schema evolution or version upgrades

### Configuration

The admin dashboard requires Turso database credentials and PostHog API access:


### Production Monitoring

Monitor these in the Vercel dashboard or via logs:

- **Build logs** — schema repair runs during deployment; look for "Schema initialized" or "Schema repaired" messages
- **Database quota** — Turso usage metrics (rows, queries, storage)
- **PostHog query errors** — If Analytics panel shows errors, check PostHog API key and `query:read` scope
- **Lighthouse scores** — Admin dashboard performance tracked in production

### Troubleshooting

**Symptom: "Schema is already healthy" but analytics not showing**
- Verify `TURSO_DATABASE_URL` and `TURSO_AUTH_TOKEN` are set in Vercel
- Check Turso dashboard for database status and connection limits
- Confirm PostHog event capture is enabled in production

**Symptom: "Cannot auto-repair: data exists" when deploying**
- This is intentional—repair is blocked if tables have data
- If you need to repair a live database, contact your admin or use Turso's backup/restore feature
- For development/staging, drop the database and redeploy

**Symptom: Analytics panel shows setup state instead of charts**
- This is normal if PostHog env vars are unset
- Set `POSTHOG_PERSONAL_API_KEY` and `POSTHOG_PROJECT_ID` in Vercel environment variables
- Wait for the next build/deployment to propagate changes
- Check that the PostHog key has `query:read` scope (server-only, not the public token)

**Symptom: Admin authentication not working**
- Verify `ADMIN_PASSWORD` is set in Vercel (production only; dev mode uses empty password)
- Check browser dev tools for `admin_session` cookie; should be HttpOnly and Secure
- Verify `SESSION_SIGNING_KEY` is set and consistent across deployments
- Clear cookies and retry login

## Security Headers

The site configures security headers in `next.config.mjs:22-35`:

```javascript
async headers() {
  return [
    {
      source: '/:path*',
      headers: [
        { key: 'X-DNS-Prefetch-Control', value: 'on' },
        { key: 'X-Frame-Options', value: 'SAMEORIGIN' },
        { key: 'X-Content-Type-Options', value: 'nosniff' },
        { key: 'Referrer-Policy', value: 'strict-origin-when-cross-origin' },
        { key: 'Permissions-Policy', value: 'camera=(), microphone=(), geolocation=()' },
      ],
    },
  ]
}
```

| Header | Value | Purpose |
| :--- | :--- | :--- |
| `X-DNS-Prefetch-Control` | `on` | Enables DNS prefetching for external resources |
| `X-Frame-Options` | `SAMEORIGIN` | Prevents clickjacking attacks (disallows embedding in iframes) |
| `X-Content-Type-Options` | `nosniff` | Prevents MIME type sniffing |
| `Referrer-Policy` | `strict-origin-when-cross-origin` | Controls referrer information sent to external sites |
| `Permissions-Policy` | `camera=(), microphone=(), geolocation=()` | Disables browser APIs that aren't used |

**Note:** CSP (Content Security Policy) is not configured because it conflicts with inline styles from TSParticles and Framer Motion. If adding CSP, use `unsafe-inline` or nonces.

## Build Optimizations

### Image Optimization

Next.js image optimization is configured in `next.config.mjs:8-13`:

```javascript
images: {
  formats: ['image/avif', 'image/webp'],
  remotePatterns: [],
  deviceSizes: [640, 750, 828, 1080, 1200, 1920, 2048],
  imageSizes: [16, 32, 48, 64, 96, 128, 256],
}
```

- **Modern formats** — serves AVIF/WebP with JPEG fallback
- **Responsive images** — generates srcset for all device sizes
- **No remote patterns** — all images are local (no external image domains allowed)

### MDX Plugins

The MDX pipeline includes 35+ remark/rehype plugins for performance:

- **Code highlighting** — `rehype-prism-plus` for syntax highlighting at build time (no runtime JS)
- **Math rendering** — `rehype-katex` for server-side LaTeX rendering
- **Link validation** — `remark-validate-links` catches broken internal links at build time
- **Reading time** — `rehype-infer-reading-time-meta` computes reading time during compilation

See [Content > MDX Pipeline](/docs/content/mdx-pipeline) for the full plugin list.

### Webpack Optimizations

Custom webpack config in `next.config.mjs:14-21`:

```javascript
webpack: (config) => {
  config.resolve.fallback = {
    ...config.resolve.fallback,
    fs: false,
    path: false,
  }
  return config
}
```

- **Disables Node.js modules** — `fs` and `path` are not bundled for client (reduces bundle size)
- **Tree shaking** — Next.js automatically tree-shakes unused exports

All admin provider secrets must stay server-only in Vercel. Do not put these values in `.env*` files committed to the repository.

Check:

- Point DNS A/CNAME records to Vercel
- Vercel automatically provisions SSL via Let's Encrypt
- Enable "Automatically expose System Environment Variables" in Vercel settings

Push to `master` branch — Vercel automatically builds and deploys.

**Vercel build command:**


## Canary + Rollback Drill

Run a local dry-run drill (no production credentials required):

```bash
pnpm drill:canary --scenario rollback
```

This produces a verifiable artifact at `.cache/drills/canary-rollback-<timestamp>.json` with checkpoint results, observed metrics, and the selected decision path.

Rollback triggers used by the drill:

- `error_rate_pct > 1`
- `p95_latency_ms > 1200`
- `success_rate_pct < 99.5`

Primary rollback command path:

```bash
vercel rollback $LAST_GOOD_DEPLOYMENT_URL --yes
vercel alias set $LAST_GOOD_DEPLOYMENT_URL $SITE_DOMAIN
vercel alias rm canary.$SITE_DOMAIN
```

## Deployment Workflow

```
1. Push to master branch
   ↓
2. Vercel triggers build
   ↓
3. Install dependencies (pnpm install)
   ↓
4. Run prebuild hook (preprocessing)
   ↓
5. Run build (next build)
   ↓
6. Deploy to CDN
   ↓
7. Invalidate cache
   ↓
8. Update DNS
```

**Build time:** Typically 2-4 minutes depending on post count and image optimization.

## Monitoring

### Performance

Use Vercel Analytics, Speed Insights, and PostHog to monitor:

- **Core Web Vitals** — LCP, FID, CLS
- **Page load times** — TTFB, FCP, TTI
- **Bundle size** — track over time via `ANALYZE=true pnpm build`
- **Visitor behavior** — top pages, referrers, searches, shares, outbound clicks, and reading progress

### Errors

Vercel automatically captures:

- **Build errors** — logged to Vercel dashboard
- **Runtime errors** — logged to Vercel Functions logs (for API routes)
- **Edge function errors** — logged to Vercel Edge logs

For client-side error monitoring, consider integrating Sentry or similar.
