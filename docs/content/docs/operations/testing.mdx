## Overview

The repo's active quality gates are centered on:

- **Vitest** for automated tests
- **ESLint** for static analysis
- **TypeScript** type checking
- **Next.js production builds**
- **Lighthouse CI** for performance/accessibility/SEO checks
- a **GitHub Actions quality workflow** that runs the main gates in CI

This page reflects the current codebase and intentionally avoids documenting tooling that is not actually wired into `package.json`.

## Vitest

The root test command is:

```bash
pnpm test
```

That runs `vitest run`.

### Current config

`vitest.config.ts` currently uses:

- default `node` environment
- `jsdom` overrides for component/browser-like tests
- `15_000ms` test timeout
- `**/*.test.{ts,tsx}` include pattern

### Current suite size

A recent local run in this repo snapshot completed:

- **17 test files**
- **245 tests**
- **9.75s duration**

Treat those numbers as a point-in-time check, not a guaranteed constant.

### Coverage

There is **no configured coverage provider** in `vitest.config.ts`, no dedicated coverage script in `package.json`, and no coverage dependency wired in the root app.

So if you need coverage reporting, that work still needs to be added first.

## ESLint

Run linting with:

```bash
pnpm lint
```

The project uses ESLint flat config with Next.js, React, React Hooks, TypeScript, and MDX-related rules in the main app config.

## Type Checking

Run:

```bash
pnpm typecheck
```

The root script is `tsc --noEmit`.

## Build Verification

Run:

```bash
pnpm build
```

This verifies the main Next.js application production build.

## TypeScript Scripts: `tsx`, Not `ts-node`

Repository scripts are executed with `tsx`.

Examples from `package.json`:

- `pnpm preprocess`
- `pnpm new-post`
- `pnpm validate:blog-manager`
- `pnpm generate-icons`

The docs should not describe a `ts-node` workflow here, because that is not what the repo currently uses.

## Lighthouse CI

Lighthouse CI is configured in `lighthouserc.cjs`.

```js
module.exports = {
  ci: {
    collect: {
      startServerCommand: "pnpm start",
      startServerReadyPattern: "Ready in",
      url: ["http://localhost:3000", "http://localhost:3000/blog"],
      numberOfRuns: 1,
    },
    assert: {
      assertions: {
        "categories:performance": ["warn", { minScore: 0.6 }],
        "categories:accessibility": ["error", { minScore: 0.8 }],
        "categories:best-practices": ["warn", { minScore: 0.8 }],
        "categories:seo": ["warn", { minScore: 0.8 }],
      },
    },
  },
};
```

### Current thresholds

| Category | Level | Minimum score |
| :--- | :--- | :--- |
| Performance | `warn` | `0.6` |
| Accessibility | `error` | `0.8` |
| Best Practices | `warn` | `0.8` |
| SEO | `warn` | `0.8` |

### Current audited URLs

- `/`
- `/blog`

### Running Lighthouse locally

The workflow installs LHCI explicitly, so the safest local equivalent is:

```bash
pnpm build
npx @lhci/cli@0.14.x autorun --config=./lighthouserc.cjs
```

There is no root `pnpm lighthouse` script right now.

## GitHub Actions

### Lighthouse workflow

`.github/workflows/lighthouse.yml` currently:

- runs on `push` and `pull_request`
- uses `ubuntu-24.04`
- reads Node from `.nvmrc`
- installs with `pnpm install --frozen-lockfile`
- builds the site
- runs `lhci autorun --config=./lighthouserc.cjs`
- uploads `.lighthouseci/**` as an artifact

### CI Quality Gates workflow

`.github/workflows/deploy.yml` defines the main quality gate job. It currently runs:

1. dependency install
2. DB migration preflight guard
3. `pnpm lint`
4. `pnpm typecheck`
5. `pnpm build`
6. Playwright browser install
7. a Playwright smoke gate step
8. artifact upload for Playwright results

## Playwright Status

The CI workflow includes a Playwright smoke step, but this repository snapshot does **not** currently expose:

- a checked-in root Playwright config file
- a root `e2e` script in `package.json`

So the honest state is:

- **CI expects a Playwright smoke gate**
- **local Playwright execution is not fully codified from the root package right now**

If you need to rely on Playwright locally, inspect the current CI workflow first and expect additional setup work.

## What Is Not Currently Documented as Active Tooling

### Storybook

This repo snapshot does not include root Storybook scripts or dependencies in `package.json`, so Storybook should not be treated as part of the current active testing stack.
