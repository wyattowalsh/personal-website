## Architectural Philosophy

The codebase is organized around a simple idea: **keep content, runtime concerns, and presentation separate enough that each can evolve without dragging the others around**.

In practice that means:

- Next.js App Router handles routing, layouts, metadata, and API boundaries
- `lib/` owns shared configuration, validation, metadata helpers, preprocessing, and backend-facing logic
- `content/posts/` remains the source of truth for published writing
- `docs/` is a separate Fumadocs workspace that documents the app without being tangled into the main runtime

That split keeps the site maintainable while still allowing rich client-side UI on the homepage and blog surfaces.

## Monorepo Structure

The project uses a **pnpm workspace** monorepo containing two packages:

```yaml
# pnpm-workspace.yaml
packages:
  - '.'          # Main Next.js application
  - 'docs'       # Fumadocs documentation site
```

This structure allows the documentation to live alongside the main application while maintaining separate dependencies and build processes.

## High-Level Directory Structure

The repository is feature-oriented, but the major boundaries are now clearer if you read it as **app runtime + shared libraries + content + docs workspace**.

```bash
personal-website/
├── app/                     # Main Next.js app routes, layouts, metadata, API handlers
│   ├── api/                 # HTTP endpoints used by the site and admin surfaces
│   ├── admin/               # Admin UI routes
│   ├── blog/                # Blog index, tags, archive, and post routes
│   ├── layout.tsx           # Global shell, providers, metadata, analytics
│   └── page.tsx             # Home page entry point
├── components/              # Reusable React components and client-side UI building blocks
│   ├── hooks/               # Shared hooks such as reduced-motion handling
│   ├── particles/           # Generated particle preset manifest + helpers
│   └── ui/                  # Base shadcn/ui primitives
├── content/
│   └── posts/               # MDX blog posts (one directory per slug)
├── lib/                     # Shared runtime code for config, metadata, content processing, feeds, etc.
│   ├── core.ts              # Config, validation schemas, API middleware, ApiError
│   ├── server.ts            # BackendService singleton and post preprocessing/search
│   ├── metadata.ts          # Metadata builders for pages and posts
│   ├── feed.ts              # RSS/Atom/JSON feed helpers
│   ├── schema.ts            # Shared structured-data helpers/schemas
│   └── utils.ts             # Small shared utilities (`cn`, formatting, MDX stripping)
├── public/                  # Static assets: images, fonts, particle JSON, favicons
├── scripts/                 # Preprocessing and maintenance scripts
└── docs/                    # Separate Fumadocs documentation app
    ├── app/                 # Docs-site routes/layout
    ├── content/docs/        # Documentation MDX files
    ├── lib/                 # Docs-site helpers/source loading
    └── package.json         # Separate dependencies and scripts for the docs app
```

## Current Runtime Layers

The repository no longer reads best as a strict three-layer `lib/` stack. A more accurate mental model is:

### 1. App Router + UI Shell (`app/`, `components/`)

- `app/layout.tsx` defines the global shell: fonts, theme provider, tooltip context, header, analytics, and structured data
- Route-specific layouts in `app/blog/` and `app/blog/posts/` add blog chrome, scroll management, math support, and post-specific structure
- Server components fetch data close to the route entry points, then hand off to client components for animation-heavy experiences like the landing page hero and particle background

### 2. Content Processing + Search (`content/posts/`, `lib/server.ts`)

- `BackendService` is the central content access layer
- It preprocesses MDX posts from `content/posts/**/index.mdx`
- Frontmatter is validated with Zod before posts enter the in-memory maps and search index
- The service also derives reading time, cleans MDX syntax for search, builds sorted/tagged caches, and exposes helpers such as `getAllPosts()`, `getPostMetadata()`, `getAdjacentPosts()`, and `getRelatedPosts()`

### 3. Admin Dashboard (`app/admin/`, `app/admin/lib/`)

- The `/admin` surface is a protected dashboard for analytics, content management, and growth insights
- It requires authentication with signed session cookies and optional rate limiting
- Admin routes connect to PostHog Query API for aggregate visitor analytics and Turso SQLite for time-series rollup storage
- The admin UI is built with reusable modular components: `MetricCard`, `DataGrid`, `InsightCard`, `DashboardHeader`, and specialized panels for Analytics, Content, Blog Stats, and Growth
- See [Operations > Admin Dashboard](/docs/operations/admin-dashboard) for full architecture and development guide

### 4. Shared Runtime Infrastructure (`lib/core.ts`, `lib/metadata.ts`, `lib/feed.ts`, `lib/logger.ts`, `lib/schema.ts`)

- `lib/core.ts` owns config, API validation schemas, `ApiError`, and the request-validation / correlation-id middleware used by route handlers
- `lib/metadata.ts` centralizes metadata generation
- `lib/feed.ts` powers feed routes
- `lib/logger.ts`, `lib/constants.ts`, and `lib/utils.ts` provide the smaller but important plumbing that keeps the app consistent

### 5. Documentation Workspace (`docs/`)

- The docs live in a separate Next.js app built with Fumadocs
- That workspace has its own `package.json`, routing tree, and content source
- Keeping it separate lets the main app and the docs site evolve independently while still living in the same repository

## Why This Split Works

- **Content stays portable**: posts live outside route code
- **UI stays composable**: layouts and components can change without rewriting data access
- **Backend logic stays testable**: preprocessing and API helpers are plain TypeScript modules
- **Docs stay close to the code**: the Fumadocs workspace can describe the current architecture without being coupled to the production app bundle
