title: "MDX Pipeline"
description: "How blog post MDX is loaded, parsed, transformed, and rendered in the main site."
---

The blog does **not** use `@next/mdx` page compilation in `next.config.mjs`. Instead, each post is read from `content/posts/{slug}/index.mdx` and compiled in the blog post route with `next-mdx-remote/rsc`.

## Where MDX Is Processed

The runtime pipeline lives in `app/blog/posts/[slug]/page.tsx`:

```tsx
const { content } = await compileMDX({
  source,
  components: mdxComponents,
  options: {
    parseFrontmatter: true,
    mdxOptions: {
      remarkPlugins: [
        remarkGfm,
        remarkMath,
        [remarkMdxMathEnhanced, { component: "Math" }],
      ],
      rehypePlugins: [
        rehypeSlug,
        [rehypePrismPlus, { ignoreMissing: true }],
        rehypeCallouts,
      ],
    },
  },
});
```

`next.config.mjs` intentionally keeps `pageExtensions` to `js`, `jsx`, `ts`, and `tsx`, so `.mdx` files are not treated as route files.

## End-to-End Flow

```text
content/posts/{slug}/index.mdx
        │
        ├─ read from disk in app/blog/posts/[slug]/page.tsx
        ├─ compileMDX(..., parseFrontmatter: true)
        ├─ remark transforms markdown/MDX syntax
        ├─ rehype transforms generated HTML AST
        ├─ mdx-components.tsx maps elements/components
        └─ rendered React content + <ArticleTracker />
```

## Frontmatter Handling

There are two separate frontmatter paths in this codebase:

### Render-time frontmatter parsing

`compileMDX({ parseFrontmatter: true })` reads frontmatter so the MDX compiler can safely process the file content.

### Backend indexing and metadata parsing

`BackendService` in `lib/server.ts` separately reads the same file with `gray-matter`, validates frontmatter with Zod, and stores the parsed result for:

- post listings
- tags
- search
- feed generation
- route metadata generation

That means frontmatter is shared, but parsing is done in two different places for two different jobs.

## Remark Plugins

These run before the Markdown/MDX tree becomes HTML.

### `remark-gfm`

Adds GitHub Flavored Markdown support such as:

- tables
- task lists
- autolinks
- strikethrough
- footnotes

### `remark-math`

Detects inline and display LaTeX syntax.

### `remark-mdx-math-enhanced`

Transforms math nodes into the project's `<Math>` component shape so equations can use the site's custom math rendering and numbering behavior.

## Rehype Plugins

These run after Markdown has been converted into HTML/JSX AST nodes.

### `rehype-slug`

Adds IDs to headings for deep linking and heading-link UI.

### `rehype-prism-plus`

Applies Prism-based syntax highlighting to fenced code blocks.

This route config passes `{ ignoreMissing: true }`, which keeps rendering from crashing when Prism does not recognize a language.

### `rehype-callouts`

Transforms supported blockquote/callout patterns into richer callout markup used by the content styles.

## Component Registry

After the plugin chain, the content is rendered through `mdx-components.tsx`.

That registry is responsible for:

- replacing standard HTML elements with styled site components
- exposing custom components such as `<Callout>`, `<Math>`, `<Mermaid>`, and `<Gist>`
- exposing selected UI primitives to MDX without local imports

See [MDX Components](/docs/content/mdx-components) for the component surface.

## Math and Diagram Notes

### Math

Math support is a collaboration between:

- `remark-math`
- `remark-mdx-math-enhanced`
- the `Math` component
- KaTeX styling in `app/globals.css`

### Mermaid

Mermaid is not part of the compiler plugin chain. Instead, the `Mermaid` component lazy-loads the `mermaid` package on the client with:

```tsx
const mermaid = (await import("mermaid")).default;
```

That keeps the heavy diagram runtime out of pages that never render Mermaid content.

## Caching and Performance

### What is cached

`BackendService` caches:

- parsed posts
- sorted post lists
- tagged post lists
- the Fuse search index

### What is not cached here

The docs should not imply that rendered MDX output is cached by `BackendService`. Its cache is for backend post metadata and search structures, not the compiled React result from `compileMDX()`.

## Changing the Pipeline

If you need to add or remove MDX plugins, update `app/blog/posts/[slug]/page.tsx`, not `next.config.mjs`.

After pipeline changes:

1. run `pnpm dev` and open a real post
2. test the content features you changed
3. run `pnpm test`
4. run `pnpm build`
