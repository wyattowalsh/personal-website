Blog posts live in `content/posts/{slug}/index.mdx`. The MDX file is the source of truth for both the article body and its frontmatter metadata.

## Quick Start

Create a new post with the bundled CLI:

```bash
pnpm new-post
```

Or create one non-interactively:

```bash
pnpm new-post \
  --title "My Post Title" \
  --tags "React,TypeScript" \
  --summary "What this post is about" \
  --slug "my-post-title"
```

The script creates:

- `content/posts/{slug}/`
- `content/posts/{slug}/index.mdx`
- starter frontmatter with today's date
- starter heading content

## Post Location and URL Shape

- **Source file:** `content/posts/{slug}/index.mdx`
- **Route:** `/blog/posts/{slug}`
- **Generator script:** `scripts/new-post.ts`

There is no per-post `page.mdx`, per-post `metadata` export, or per-post JSON-LD component in the MDX file.

## Frontmatter Reference

The backend validates frontmatter with Zod in `lib/server.ts`.

```yaml
---
title: "Post Title"
image: "/post-hero.svg"
caption: "Optional short tagline"
summary: "Optional SEO/social description"
tags: ["Category", "Technology"]
created: "2026-01-15"
updated: "2026-01-15"
series:
  name: "Optional Series Name"
  order: 1
---
```

### Fields

| Field | Type | Required | Notes |
|-------|------|----------|-------|
| `title` | `string` | Yes | Display title, feed title, metadata title |
| `created` | `string` | Yes | Publication date, expected in `YYYY-MM-DD` form |
| `updated` | `string` | No | Falls back to `created` when omitted |
| `image` | `string` | No | Absolute path under `public/`, or a full URL |
| `caption` | `string` | No | Short subtitle/tagline used in post UI |
| `summary` | `string` | No | Used for metadata, feeds, and search summaries |
| `tags` | `string[]` | No | Defaults to `[]` |
| `series` | `{ name: string; order: number }` | No | Lets related posts be grouped and ordered |

### Practical guidance

- Keep `title` specific and skimmable.
- Prefer `summary` when you want better feed cards and metadata descriptions.
- Reuse existing tags where possible so tag pages stay coherent.
- Store hero images in `public/` and reference them as `/filename.ext`.
- Use `series` only when readers should consume posts in order.

## Metadata and SEO Flow

Frontmatter is the source of truth, but metadata is generated centrally by the app:

1. **Indexing and feeds**  
   `BackendService` reads `content/posts/**/index.mdx`, parses frontmatter with `gray-matter`, validates it with Zod, and builds the post/search caches.

2. **Page rendering**  
   `app/blog/posts/[slug]/page.tsx` loads the MDX file and compiles it with `compileMDX({ parseFrontmatter: true })`.

3. **Route metadata**  
   `app/blog/posts/[slug]/layout.tsx` calls `generatePostMetadata()` from `lib/metadata.ts`.

4. **Structured data**  
   The same layout injects JSON-LD from `generatePostStructuredData()`.

That means the MDX file itself should contain frontmatter and article content only.

## CLI Reference

`pnpm new-post` supports two modes:

### Interactive mode

Run with no flags to be prompted for:

- title
- slug
- summary
- tags
- caption

### CLI mode

Supported flags:

| Flag | Description |
|------|-------------|
| `--title` | Post title |
| `--slug` | Slug override |
| `--summary` | Summary text |
| `--tags` | Comma-separated tags |
| `--caption` | Optional caption |
| `--image` | Hero image path override |

If `--slug` is omitted, the script slugifies the title.

## Publishing Workflow

1. Run `pnpm new-post`
2. Add or update hero assets in `public/` if needed
3. Write the post in `content/posts/{slug}/index.mdx`
4. Preview at `http://localhost:3458/blog/posts/{slug}`
5. Run `pnpm lint && pnpm typecheck && pnpm test`
6. Run `pnpm build` before shipping

## Post Discovery and Indexing

`BackendService` in `lib/server.ts` automatically:

1. scans `content/posts/**/index.mdx`
2. parses and validates frontmatter
3. strips MDX syntax for search/index text
4. computes reading time and word count
5. builds the in-memory tag list and Fuse search index

No manual registration step is required.
