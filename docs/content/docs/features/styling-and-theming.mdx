The main site uses **Tailwind CSS v4** with a CSS-first setup:

- `app/tailwind.css` defines theme tokens and animation utilities with `@theme`
- `app/globals.css` defines the underlying CSS variables and global styles
- `next-themes` provides dark-mode switching at runtime

## Core Files

| File | Purpose |
|------|---------|
| `app/tailwind.css` | Tailwind v4 entry point, `@theme` tokens, animation names, keyframes |
| `app/globals.css` | Root theme variables, prose styles, math styles, global utilities, reduced-motion rules |
| `app/layout.tsx` | Mounts the root `ThemeProvider` |
| `lib/utils.ts` | `cn()` helper for composing class names |

## Styling Model

The project intentionally splits responsibilities:

1. **Tailwind utilities** for layout, spacing, typography, and responsive behavior
2. **CSS custom properties** for themeable values shared across light and dark mode
3. **Global CSS rules** for prose, math, code blocks, and other content-heavy surfaces

## Tailwind v4 `@theme` Mapping

`app/tailwind.css` maps the CSS variables into Tailwind tokens:

```css
@theme {
  --color-background: hsl(var(--background));
  --color-foreground: hsl(var(--foreground));
  --color-primary: hsl(var(--primary));
  --color-primary-foreground: hsl(var(--primary-foreground));
  --color-muted: hsl(var(--muted));
  --color-accent: hsl(var(--accent));
  --color-card: hsl(var(--card));
  --radius-lg: var(--radius);
  --shadow-glow: var(--shadow-glow);
}
```

This replaces the old `tailwind.config.js`-style theme extension pattern.

## Global CSS Responsibilities

`app/globals.css` carries a lot of the real UI behavior:

- prose defaults
- inline code and fenced code styling

Because this is a content-heavy site, `globals.css` is part of the design system, not just a reset file.

## Dark Mode Runtime

Dark mode is provided by `next-themes` in `app/layout.tsx`:

```tsx
<ThemeProvider
  attribute="class"
  defaultTheme="system"
  enableSystem
  disableTransitionOnChange
>
  {children}
</ThemeProvider>
```

### Important details

- the root provider uses the `class` attribute strategy
- the default theme is the user's system preference
- theme transitions are suppressed during flips with `disableTransitionOnChange`

### Reading the active theme

Client components can read the active theme with `useTheme()`:

```tsx
import { useTheme } from "next-themes";

const { theme, setTheme, resolvedTheme } = useTheme();
```

`resolvedTheme` is a **client-side** value. The docs should not imply that server components directly read it.

One concrete example is `components/Mermaid.tsx`, which switches Mermaid's runtime theme based on `resolvedTheme`.

## `cn()` Utility

Use `cn()` from `lib/utils.ts` when composing class names:

```tsx
import { cn } from "@/lib/utils";

<div
  className={cn(
    "rounded-lg border",
    isActive && "border-primary text-primary",
    compact ? "p-3" : "p-6"
  )}
/>
```

It combines `clsx` for conditional logic and `tailwind-merge` for conflict resolution.

## Accessibility

### Reduced motion

`app/globals.css` includes a global `prefers-reduced-motion` override:

```css
@media (prefers-reduced-motion: reduce) {
  *,
  *::before,
  *::after {
    animation-duration: 0.01ms !important;
    animation-iteration-count: 1 !important;
    transition-duration: 0.01ms !important;
    scroll-behavior: auto !important;
  }
}
```

### Focus styles

Interactive components consistently rely on focus-visible ring patterns using `ring`, `ring-offset`, and related Tailwind utilities.

## Practical Guidance

1. Reach for Tailwind utilities first
2. Put reusable theme values in CSS variables, not hard-coded colors
3. Treat `globals.css` as the home for cross-cutting content styles
4. Keep theme-dependent logic in client components when it needs `resolvedTheme`
5. Use `cn()` instead of string concatenation for conditional Tailwind classes
