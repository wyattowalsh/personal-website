## The Problem

<Callout type="warning">
Agent workflows get brittle when the instructions, tools, and review habits that make them useful are copied by hand from one runtime to another.
</Callout>

The failure mode is familiar: I improve a review workflow in Claude Code, then forget to update the Codex version. I tune a docs prompt in one repo, but the global instruction file still says something older. I fix an install command, but the next agent run finds the stale one first.

<Mermaid chart={`
flowchart LR
    Idea[Useful workflow] --> Copy[Copy into one tool]
    Copy --> Fork[Another tool needs it]
    Fork --> Patch[Patch by hand]
    Patch --> Drift[Versions drift]
    Drift --> Rediscover[Rediscover the same fix]
    Rediscover --> Patch

    style Drift fill:#fee2e2,stroke:#ef4444
    style Rediscover fill:#fff7ed,stroke:#f97316
`} title="Prompt and config drift" />

That is manageable for one prompt. It breaks down when the setup becomes a working system: review skills, MCP server builders, docs stewards, language conventions, install targets, hooks, evals, and public docs.

Agents is my answer to that problem: make the agent layer durable enough to review, validate, install, and update like software.

## What Agents Does

<Callout type="success">
Agents packages reusable agent workflows as source-controlled skills, instructions, MCP configuration, plugin metadata, docs, and CLI tooling.
</Callout>

The smallest useful unit is a skill: a focused workflow with frontmatter, invocation guidance, references, hooks, evals, or helper scripts when the workflow needs them. The repo currently carries 47 skills, covering code review, MCP creation, release planning, docs stewardship, shell conventions, and prompt engineering.

The point is not the exact count. The point is that a workflow becomes a file: reviewable, testable, linkable, installable, and independent of hidden tool state.

## How I Use It

Agents is not meant to be a bookshelf of prompts. It is a working loop: pick the right skill, run the workflow, improve the asset when reality proves it weak, validate the repo, then reinstall or publish from the same source.

<Mermaid chart={`
flowchart LR
    Task[Real task] --> Route[Choose a skill]
    Route --> Run[Run inside an agent]
    Run --> Learn[Find a gap]
    Learn --> Patch[Refine the repo asset]
    Patch --> Validate[Validate with wagents]
    Validate --> Docs[Regenerate docs]
    Docs --> Install[Install or update]
    Install --> Run

    style Patch fill:#eef2ff,stroke:#4f46e5
    style Validate fill:#f0fdf4,stroke:#16a34a
`} title="The agent workflow loop" />

That loop is why the repo includes more than `SKILL.md` files. A useful workflow usually needs source references, deterministic checks, docs, install commands, and a path back into the runtimes where I actually use it.

| Step | Example |
| --- | --- |
| Route | Use [`honest-review`](https://agents.w4w.dev/skills/honest-review/) for a review pass. |
| Refine | Update a skill reference, hook, or eval when the workflow misses something. |
| Validate | Run `wagents validate`, eval checks, hook checks, and docs checks. |
| Publish | Reinstall with `npx skills add` or ship through plugin metadata. |

## Architecture

At a high level, Agents is a repo-first distribution pipeline. Source files feed `wagents`; `wagents` enforces and renders the catalog; docs, packages, plugins, and install commands expose the result to agent runtimes.

## Validation and Release Discipline

Agent assets fail differently than application code. A bad skill can still be valid Markdown. A stale install command can still look plausible. A hook can drift from the runtime event names it expects.

That is why `wagents` is part of the project, not just a convenience script.

<CodeFilename>snapshot-validator.json</CodeFilename>

```json
{
  "ok": true,
  "error_count": 0,
  "errors": [],
  "message": "All validations passed"
}
```

The repo also runs those checks at the boundary.

| Gate | What it proves | Reference |
| --- | --- | --- |
| Code checks | Ruff, Ty, and pytest keep the CLI and helpers healthy. | [CI](https://github.com/wyattowalsh/agents/blob/main/.github/workflows/ci.yml) |
| Catalog checks | `wagents validate` catches broken skill, agent, hook, and relationship metadata. | [CLI docs](https://agents.w4w.dev/cli/) |
| Docs checks | README freshness, docs generation, Astro checks, and docs build keep public surfaces aligned. | [CI](https://github.com/wyattowalsh/agents/blob/main/.github/workflows/ci.yml) |
| Release checks | Strict skill audit and packaging verification run before tagged release artifacts. | [release workflow](https://github.com/wyattowalsh/agents/blob/main/.github/workflows/release-skills.yml) |

<Details summary="Validation command map">

```bash
# Check repository assets
uv run wagents validate

# Check eval definitions
uv run wagents eval validate

# Check hook metadata
uv run wagents hooks validate

# Check generated README freshness
uv run wagents readme --check

# Package a skill without publishing it
uv run wagents package skill-creator --dry-run

# Build public docs
pnpm --dir docs build
```

</Details>

That validation stack is what moves the project from one-off prompts toward maintainable agent infrastructure.

## Trade-offs

Agents is still early. The public GitHub metadata did not show a latest release when I last checked this post, and some surfaces are intentionally reserved for future expansion. The repo-level `agents/` directory, for example, is reserved for bundled agent definitions; the current emphasis is skills, shared instructions, MCP configuration, plugin metadata, docs, and CLI tooling.

Exact counts also age. The catalog had 47 skills at this snapshot, and the docs described 30 configured MCP tools from `mcp.json`. Those numbers are useful orientation, not the durable claim.

The durable claim is simpler: reusable agent workflows need a durable home in git.
