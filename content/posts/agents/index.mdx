## The Problem

<Callout type="warning">
Agent workflows get brittle when the instructions, tools, and review habits that make them useful are copied by hand from one runtime to another.
</Callout>

The failure mode is familiar: I improve a review workflow in Claude Code, then forget to update the Codex version. I tune a docs prompt in one repo, but the global instruction file still says something older. I add a safer install command, but the next agent run finds the stale one first.

<Mermaid chart={`
flowchart LR
    Idea[Useful workflow] --> Copy[Copy into one tool]
    Copy --> Fork[Another tool needs it]
    Fork --> Patch[Patch by hand]
    Patch --> Drift[Versions drift]
    Drift --> Rediscover[Rediscover the same fix]
    Rediscover --> Patch

    style Drift fill:#fee2e2,stroke:#ef4444
    style Rediscover fill:#fff7ed,stroke:#f97316
`} title="Prompt and config drift" />

That is manageable for one prompt. It breaks down when the setup becomes a working system: review skills, prompt-engineering workflows, MCP server builders, docs stewards, language conventions, install targets, hooks, evals, and public docs.

Agents is my answer to that problem: make the agent asset layer boring enough to review, validate, install, and update like software.

---

## What Agents Does

<Callout type="success">
Agents packages reusable agent workflows as source-controlled skills, instructions, MCP configuration, plugin metadata, docs, and CLI tooling.
</Callout>

The smallest useful unit is a skill: a focused workflow with frontmatter, invocation guidance, references, hooks, evals, or helper scripts when the workflow needs them. The repo currently carries 47 skills, covering everything from code review to MCP creation, release planning, docs stewardship, shell conventions, and prompt engineering.

The important part is not the exact count. The important part is that the workflow is a file. It can be reviewed, tested, linked, installed, and improved without relying on hidden tool state.

## Architecture

At a high level, Agents is a repo-first distribution pipeline. Source files feed `wagents`; `wagents` validates and renders the catalog; docs and install paths expose the same source to different agent runtimes.

## Skill Catalog

The catalog is intentionally broad, but the pattern is consistent: each skill names the work it owns, says when to use it, says when not to use it, and can carry references, helper scripts, hooks, or evals when plain instructions are not enough.

<Callout type="info">
I want skills to behave more like small operational packages than pasted prompts: inspectable, documented, versioned, and validated.
</Callout>

<Details summary="Source files behind the catalog">

- [`skills/`](https://github.com/wyattowalsh/agents/tree/main/skills) contains repo-owned skill definitions.
- [`docs/src/content/docs/skills/`](https://github.com/wyattowalsh/agents/tree/main/docs/src/content/docs/skills) contains generated and curated public skill pages.
- [`wagents/catalog.py`](https://github.com/wyattowalsh/agents/blob/main/wagents/catalog.py) scans skills, agents, MCP servers, and relationships.
- [`wagents/cli.py`](https://github.com/wyattowalsh/agents/blob/main/wagents/cli.py) exposes validation, docs, hooks, evals, package, install, and skill-index commands.
- [`skills/skill-creator/scripts/audit.py`](https://github.com/wyattowalsh/agents/blob/main/skills/skill-creator/scripts/audit.py) audits skills for structure and quality gates.

</Details>

That source layout matters because the catalog can grow without turning into a private notebook. A new skill can be added, documented, linked to related skills, validated, and packaged from the same repo.

## Validation and Release Discipline

Agent assets fail differently than application code. A bad skill can still be valid Markdown. A stale install command can still look plausible. A hook can drift from the runtime event names it expects.

That is why `wagents` is part of the project, not just a convenience script.

<CodeFilename>snapshot-validator.json</CodeFilename>

```json
{
  "ok": true,
  "error_count": 0,
  "errors": [],
  "message": "All validations passed"
}
```

The repo also runs those checks at the boundary.

| Gate | What it proves | Reference |
| --- | --- | --- |
| Ruff lint and format | Python code stays mechanically clean | [CI](https://github.com/wyattowalsh/agents/blob/main/.github/workflows/ci.yml) |
| Ty | Python types stay checked | [CI](https://github.com/wyattowalsh/agents/blob/main/.github/workflows/ci.yml) |
| pytest | CLI and helper behavior stay covered | [CI](https://github.com/wyattowalsh/agents/blob/main/.github/workflows/ci.yml) |
| `wagents validate` | Skill, agent, hook, and relationship contracts hold | [CLI docs](https://agents.w4w.dev/cli/) |
| `wagents readme --check` | Public README is not stale | [CI](https://github.com/wyattowalsh/agents/blob/main/.github/workflows/ci.yml) |
| Docs build | Generated and curated docs still compile | [CI](https://github.com/wyattowalsh/agents/blob/main/.github/workflows/ci.yml) |
| Strict skill audit | Release candidates pass the skill quality audit | [release workflow](https://github.com/wyattowalsh/agents/blob/main/.github/workflows/release-skills.yml) |

<Details summary="Validation command map">

```bash
# Check repository assets
uv run wagents validate

# Check eval definitions
uv run wagents eval validate

# Check hook metadata
uv run wagents hooks validate

# Check generated README freshness
uv run wagents readme --check

# Package a skill without publishing it
uv run wagents package skill-creator --dry-run

# Build public docs
pnpm --dir docs build
```

</Details>

That validation stack is what moves the project from "useful prompts" toward maintainable agent infrastructure.

## Trade-offs

Agents is still early. The public GitHub metadata did not show a latest release when I first wrote this post, and some surfaces are intentionally reserved for future expansion. The repo-level `agents/` directory, for example, is reserved for bundled agent definitions while the current emphasis is skills, shared instructions, MCP configuration, plugin metadata, docs, and CLI tooling.

Exact counts also age. The catalog had 47 skills at this snapshot, and the docs described 30 configured MCP tools from `mcp.json`. Those numbers are useful orientation, not the durable claim.

The durable claim is simpler: reusable agent workflows should have a source of truth.
