I rebuilt my personal website when it stopped being just a place to put links.

The homepage still has to answer the simple questions: who I am, where to find me, and what I am working on. The harder part is everything around that first screen. A real personal site has posts, tags, feeds, metadata, search, structured data, comments, analytics, operational checks, and visual experiments. If each surface owns its own copy of the truth, the site gets stale fast.

This version makes the site a small publishing system with a homepage attached.

## The Constraint

<Callout type="warning">
The main failure mode for a personal website is drift: the post body says one thing, the card preview says another, the feed says a third, and the Open Graph metadata keeps pointing at an older version.
</Callout>

I wanted the content model to be boring. A post should be one file. The post route should render that file. Search, feeds, tags, related posts, metadata, and JSON-LD should derive from the same parsed object.

That constraint shaped the rest of the site. The homepage can be playful. The particle background can change by theme. The admin area can surface operational health. But the publishing path needs to stay small enough that changing a post title does not become a scavenger hunt.

## System Shape

The core flow is:

<Mermaid chart={`
flowchart LR
    MDX["content/posts/{slug}/index.mdx"] --> Service["BackendService.preprocess"]
    Service --> Model["Post model"]
    Model --> Pages["Blog pages"]
    Model --> Tags["Tag pages"]
    Model --> Search["Fuse search"]
    Model --> Feeds["RSS / Atom / JSON Feed"]
    Model --> Metadata["Open Graph / Twitter / JSON-LD"]
    Model --> Admin["Content health"]
`} title="Personal Website content flow" />

`lib/server.ts` is the center of that pipeline. It reads `content/posts/**/index.mdx`, validates frontmatter with Zod, strips MDX syntax for plain-text consumers, calculates word count and reading time, tracks tags, builds a Fuse search index, and exposes the post collection to the app.

The route layer stays generic. `app/blog/posts/[slug]/page.tsx` compiles the MDX. `app/blog/posts/[slug]/layout.tsx` loads the parsed post, generates metadata, injects structured data, and passes adjacent, related, and series navigation into the shared post layout.

That split keeps authored content out of the app router. A post is frontmatter plus MDX, not a private route module.

## Publishing Contract

The authoring contract is intentionally plain:

```mdx
---
title: "Post Title"
image: "/post-hero.svg"
caption: "Short tagline"
summary: "One concise summary sentence."
tags: ["Project", "TypeScript"]
created: "2026-04-26"
updated: "2026-04-26"
---

Post content starts here.
```

There is no per-post `export const metadata`, no hand-written article schema, and no duplicate app-route file. The shared layout owns those concerns.

| Need | Where it comes from |
|:--|:--|
| Post page | Authored MDX compiled by the shared slug route |
| Blog cards | Parsed frontmatter and reading metadata |
| Tags | Frontmatter `tags` |
| Search | Title, summary, tags, and stripped content |
| Feeds | Shared post model rendered through `feed` |
| Social previews | `generatePostMetadata()` |
| Article schema | `generatePostStructuredData()` |

This is the part of the site I care about most. The system can grow around it, but the publishing contract should stay dull, readable, and hard to misuse.

## Homepage Identity

The homepage is built as a server/client split. `app/page.tsx` preprocesses content on the server, selects recent posts, and passes post metadata into `HomePageClient`. The client side handles the animated title, social link grid, latest writing section, and particle background.

The visible result is a link surface, but the implementation treats the blog as first-class. The latest writing section is not hard-coded. It comes from the same post model as the archive, tags, feeds, search, and individual post pages.

That matters because the homepage is usually where personal sites become stale. Here, publishing a post is enough to update the front door.

## Particle Runtime

The particle system is the most decorative part of the site, so it has the strictest boundaries.

Particles run only on the client. The runtime chooses a supported preset for the active theme, fetches the JSON config, merges density and viewport options, and falls back to a static gradient when motion is reduced, disabled, or unavailable.

<Mermaid chart={`
flowchart TB
    Theme["Resolved theme"] --> Preset["Supported preset"]
    Preset --> Config["JSON config"]
    Config --> Runtime["Runtime density options"]
    Runtime --> Canvas["tsParticles canvas"]
    Controls["Settings / pause / shuffle"] --> Runtime
    Reduced["Reduced motion or off"] --> Fallback["Static gradient"]
`} title="Particle runtime" />

The presets are grouped by mood and family: technical mesh, cosmic drift, organic motion, atmospheric fields, and a few more dramatic variants. The controls let visitors reduce density, pause motion, or switch styles without making animation a requirement for reading the page.

## Operational Surface

The site has a private admin layer because publishing systems need feedback loops. The admin area pulls together recent posts, content health, telemetry, provider snapshots, analytics, and rate-limit state. It is not part of the public reading experience, but it checks the same model the public site uses.

The public operational surfaces are standard but important:

| Surface | Purpose |
|:--|:--|
| `/feed.xml`, `/feed.atom`, `/feed.json` | Feed readers and integrations |
| `/sitemap.xml` | Search discovery |
| `/opensearch.xml` | Browser/search integration |
| route metadata | Canonical, Open Graph, and Twitter cards |
| structured data | Website, person, breadcrumb, and article schema |
| error/loading pages | Stable route behavior when content or data is missing |

The point is not to make the site feel enterprise. The point is to make it observable enough that content problems are visible before readers hit them.

## Trade-offs

This is more machinery than a static link page needs. That is deliberate.

The upside is that posts are searchable, feedable, tagged, shareable, and renderable from one source. The homepage can stay expressive without contaminating the content model. The admin area can inspect the same data that the public routes consume.

The cost is that even content changes go through a real validation loop:

```bash
pnpm preprocess
pnpm lint
pnpm typecheck
pnpm test
pnpm build
```

For this repo, that is the right trade. The website is not just a portfolio shell. It is the place where writing, project notes, identity, and small web-system experiments meet.

---

## Next

The next improvements are mostly about tightening feedback: better visual regression coverage for the homepage, clearer docs for the publishing pipeline, richer content-health checks, and cleaner examples for feeds and metadata.

The ground rule stays the same: one personal website, one authored content model, and no hidden second copy of a post.
