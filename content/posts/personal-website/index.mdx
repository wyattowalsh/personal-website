I rebuilt my personal website when it stopped being just a place to put links.

The homepage still has to answer the basics: who I am, where to find me, and what I am writing. The harder part is everything behind that first screen. A useful personal site needs posts, tags, search, feeds, previews, structured data, comments, analytics, and enough operational feedback that broken content does not quietly ship.

This version is a small publishing system with a homepage attached.

## The Constraint

<Callout type="warning">
The failure mode is drift: the post body changes, but the card preview, feed item, sitemap, Open Graph metadata, and related-post logic keep describing an older version.
</Callout>

I wanted the authoring model to stay boring. A post should be one file. The route should render that file. Every other public surface should derive from the parsed post data.

That constraint made the rest of the build easier to reason about. The homepage can be playful. The particle background can be optional. The admin area can collect operational signals. But the publishing path has to stay small enough that changing a title does not become a scavenger hunt.

## Source Map

The site is organized around a few shared ownership boundaries.

| Surface | Source of truth | Job |
|:--|:--|:--|
| Authored posts | `content/posts/{slug}/index.mdx` | YAML frontmatter plus MDX body |
| Post model | `lib/server.ts` | Parse, validate, cache, search, tags, reading time |
| Post route | `app/blog/posts/[slug]/page.tsx` | Compile authored MDX through shared components |
| Post metadata | `app/blog/posts/[slug]/layout.tsx` | Generate metadata, JSON-LD, adjacent, related, and series data |
| Homepage | `app/page.tsx` + `HomePageClient` | Pull recent posts into the front door |
| Particles | `ParticlesBackground` + particle config metadata | Client-only visual runtime with reduced-motion fallback |
| Operations | `app/admin/*` and admin libs | Content health, telemetry, provider snapshots, analytics |

The important part is not the file list. It is the direction of dependency: authored content flows outward into public surfaces, not the other way around.

## Content Flow

<Mermaid chart={`

    Model --> Pages["Blog pages"]

    Model --> Metadata["Open Graph / Twitter / JSON-LD"]
    Model --> Admin["Content health"]
`} title="Personal Website content flow" />

`BackendService.preprocess` is the center of the pipeline. It reads every authored post, validates frontmatter with Zod, strips MDX syntax for plain-text consumers, calculates word count and reading time, tracks tags, builds a Fuse search index, and exposes the post collection to routes and APIs.

The route layer stays generic. `page.tsx` compiles MDX. `layout.tsx` loads the parsed post and handles metadata, structured data, adjacent navigation, related posts, and series navigation. No individual post owns route code.

That split gives the site a useful invariant: if preprocessing accepts a post, the rest of the app consumes the same post model.

## Publishing Contract

The authored file stays intentionally plain:

```mdx
---
title: "Post Title"
image: "/post-hero.svg"
caption: "Short tagline"
summary: "One concise summary sentence."
tags: ["Project", "TypeScript"]
created: "2026-04-26"
updated: "2026-04-26"
---

Post content starts here.
```

There is no per-post `export const metadata`, no hand-written article schema, and no duplicate app-route file. The shared layout owns those concerns.

| When frontmatter changes | What updates |
|:--|:--|
| `title` | Post header, cards, feeds, metadata, adjacent/related labels |
| `summary` | Header text, search text, feed summaries, social descriptions |
| `tags` | Tag pages, search index, related-post scoring, metadata keywords |
| `created` / `updated` | Sorting, feeds, sitemap data, article schema |
| `image` / `caption` | Post header media and social preview image selection |

The contract is not fancy, but it is hard to misuse. The author edits one file, and the app decides how that post appears everywhere else.

## Homepage Identity

The homepage is deliberately split between server data and client interaction.

`app/page.tsx` ensures preprocessing has run, gets the full post list, selects the three most recent posts, and passes only metadata into `HomePageClient`. The client component handles the profile image, animated title, social link grid, latest-writing section, and particle background.

That makes the homepage feel like a link surface while still behaving like part of the publishing system. Publishing a post updates the front door without a second edit.

## Particle Runtime

The particle system is decorative, so it has tight boundaries.

It runs only on the client. It chooses a supported preset for the resolved theme, fetches the JSON config, merges runtime density options, and falls back to a static gradient when motion is reduced, disabled, or unavailable.

The presets are grouped by mood and family: technical mesh, cosmic drift, organic motion, atmospheric fields, and a few dramatic variants. Visitors can reduce density, pause motion, or switch styles. The effect adds identity, but it is never required to read the site.

## Operational Surface

The private admin area exists because a publishing system needs feedback loops. It pulls together recent posts, content health, telemetry, provider snapshots, analytics, and rate-limit state. It is not part of the public reading experience, but it checks the same content model that readers see.

The public operational surfaces stay standard and predictable:

| Surface | Purpose |
|:--|:--|
| `/feed.xml`, `/feed.atom`, `/feed.json` | Feed readers and integrations |
| `/sitemap.xml` | Search discovery |
| `/opensearch.xml` | Browser/search integration |
| route metadata | Canonical, Open Graph, and Twitter cards |
| structured data | Website, person, breadcrumb, and article schema |
| error/loading pages | Stable route behavior when content or data is missing |

The goal is not to make the site feel like enterprise software. The goal is to make content problems visible before readers find them.

## Reusable Pattern

The pattern I would reuse is simple:

1. Keep authored content in one boring format.
2. Parse it into one typed model.
3. Derive public surfaces from that model.
4. Keep decorative client runtime separate from content correctness.
5. Treat validation as part of publishing, not as a deployment afterthought.

That is the real shape of this project. The stack is Next.js, React, TypeScript, Tailwind CSS, and MDX, but the useful idea is the ownership boundary.

---

## Trade-offs

This is more machinery than a static link page needs. That is deliberate.

The upside is that posts become searchable, tagged, shareable, feed-ready, and renderable from one source. The homepage can stay expressive without contaminating the content model. The admin area can inspect the same data that the public routes consume.

The cost is that even content changes go through a real validation loop:

```bash
pnpm preprocess
pnpm lint
pnpm typecheck
pnpm test
pnpm build
```

For this repo, that is the right trade. The website is not just a portfolio shell. It is where writing, project notes, identity, and small web-system experiments meet.

## Next

The next improvements are mostly about feedback: stronger visual regression coverage for the homepage, clearer docs for the publishing pipeline, richer content-health checks, and cleaner examples for feeds and metadata.

The ground rule stays the same: one personal website, one authored content model, and no hidden second copy of a post.
